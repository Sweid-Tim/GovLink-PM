# 项目部 Obsidian 协作原则

> 本文档定义项目部在 Obsidian 知识库中的协作规范。
> 基于 GovLink 实际工作流程提炼，所有团队成员应共同遵守。

---

## 一、核心理念

### 1.1 知识即资产
每一次团组的执行、每一份方案的输出、每一个客户的反馈，都是公司的知识资产。**沉淀比执行本身更重要。** 做完一个项目不等于结束，把过程中产生的数据、经验、教训写进知识库，才是一次完整的执行。

### 1.2 SOP 驱动，不重复造轮子
能查 SOP 解决的问题，不要凭经验发挥；
能复用历史数据的问题，不要从零搜索。
**先查、再问、后做。**

### 1.3 脱敏即安全
所有对外沟通（给境外、地接、付款方）的信息，**必须对客户单位信息做脱敏处理**。仅标注团组时间和类型即可，不出现具体客户单位名称。


- 项目级文档放 `raw/sources/PM/` 对应子目录

## 三、知识库分层（"三明治"架构）

整个 vault 分为三层，理解和遵守这个分层是协作的基础：

| 层级 | 位置 | 谁维护 | 规则 |
|------|------|--------|------|
| **Raw Sources** | `raw/sources/PM/` | 所有人可写 | 原始文件不可变，LLM只读不修改 |
| **SOP** | `raw/sources/PM/SOP/` | 项目部集体维护 | 所有方案必须基于SOP输出 |
| **Knowledge** | `raw/sources/PM/Knowledge/` | 谁产出谁归档 | 项目完成后整理有价值内容入库 |
| **Historical** | `raw/sources/PM/Historical/` | 谁执行谁更新 | 酒店、餐厅、导游报表实时更新 |
| **Wiki** | `wiki/` | LLM维护 | 自动编译的持久知识制品 |
| **项目部** | `项目部/` | 每位成员 | 个人日志与团队协作入口 |

## 四、项目执行规范

### 4.1 标准执行流程

```
接单 → 查历史 → 出方案 → 内部审核 → 客户确认 → 执行 → 归档
```

每一步的产出物：
1. **航班方案** → 按 SOP 模板 + 携程/航司官网实时查询
2. **酒店方案** → Booking/携程/维途交叉比价，三公经费合规
3. **行程表** → 报批版（用词规范）+ 实际版（精确到分）
4. **出行须知** → 含12项必备内容（见 SOP-04）

### 4.2 核心红线

| 红线 | 说明 |
|------|------|
| 🚫 **不胡编乱造** | 航班价格、酒店评分、餐厅信息等必须来自真实查询 |
| 🚫 **不跳过 SOP** | 每次方案输出严格 SOP 模板，不能省略字段 |
| 🚫 **不对外暴露客户名** | 发给境外/地接/付款方时，脱敏处理客户信息 |
| 🚫 **不把敏感信息给 AI** | 政府客户敏感信息不能使用 AI 工具处理 |
| 🚫 **不改 .obsidian 配置** | 插件和设置由管理员维护，团队成员不修改 |

### 4.3 三公经费合规

- **住宿费**：一人一间，按2017版标准执行，超标200-300元以内可申请说明
- **伙食费 + 公杂费**：按定额标准发放，不需发票
- **国际旅费**：首选国内航司、经济舱、直飞

### 4.4 质量要求

- 所有方案提供 2-3 个备选，标注推荐理由
- 行程必须"全过程模拟"——从客户离开家门到回到家门
- 国际航班至少预留3小时到机场
- 境外大型活动（展会、论坛、节假日）必须在方案中排查说明
- 航班路途安排在周六/周日，确保公务落在工作日上午到下午

## 五、团队协作约定

### 5.1 信息同步
- **复杂项目**（跨多国、多部门、多人协作）→ 建立飞书共享文档同步进度
- **常规项目** → 在工作日志中记录关键节点
- **周会** → 参与并按时输出会议纪要

### 5.2 客户信息的脱敏规则

| 场景 | 做法 |
|------|------|
| 给境外地接发操作单 | 不写客户单位，写"6月法国英国团组8人" |
| 给票代出票 | 脱敏后提供名单 |
| 在 AI 工具中问问题 | 不出现客户单位名称和敏感信息 |
| 内部日志/知识库 | 可记录客户名，但注意区分密级 |

### 5.3 文件命名规范

- 文件名：`[日期]-[项目名称]-[内容类型].md`
- 示例：`2026-06-14-法国英国团组-航班方案.md`
- 禁止：中文空格、特殊符号、过长文件名

### 5.4 Wiki 更新机制

项目结束后，如需将经验沉淀为 Wiki 页面：
- 通知 LLM（Claude）处理，按 `wiki/` 结构自动生成
- 人工审核后确认入库
- 更新 [[wiki/index.md]] 和 [[wiki/log.md]]

## 六、Git 同步规范

### 6.1 自动同步
- 每天早上 8:00（北京时间）自动 commit + push 至 GitHub
- 同步脚本位于 `.claude/sync-to-github.sh`
- 不手动操作 git 命令（除非管理员要求）

### 6.2 注意事项
- `.obsidian/`、`.DS_Store`、`.claudian/sessions/` 已在 `.gitignore` 中排除
- 个人工作日志的更新会随自动同步推送到远端
- 如需紧急同步，联系管理员 Tim

### 6.3 首次加入团队
1. 注册 GitHub 账号
2. 将用户名发给 Tim
3. 安装 Git + Obsidian + Node.js
4. `git clone` 仓库到本地
5. 在 Obsidian 中打开本地文件夹

详细步骤见 [[raw/sources/PM/Knowledge/GovLink团队协作环境配置指南V1]]
