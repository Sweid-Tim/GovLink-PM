# 项目部 Obsidian 协作原则

> 本文档定义项目部在 Obsidian 知识库中的协作规范。
> 基于 GovLink 实际工作流程提炼，所有团队成员应共同遵守。

---

## 一、核心理念

### 1.1 知识即资产
每一次团组的执行、每一份方案的输出、每一个客户的反馈，都是公司的知识资产。**沉淀比执行本身更重要。** 做完一个项目不等于结束，把过程中产生的数据、经验、教训写进知识库，才是一次完整的执行。

### 1.2 SOP 驱动，不重复造轮子
能查 SOP 解决的问题，不要凭经验发挥；
能复用历史数据的问题，不要从零搜索。
**先查、再问、后做。**

### 1.3 脱敏即安全
所有前期对外沟通（给境外、地接、付款方、南京西联、供应商、酒店、航司、票代、境外机构、商协会等）的信息，**必须对客户单位信息做脱敏处理**。仅标注团组时间、客户类型、区域层级和资源需求，不出现具体客户单位名称。

例外：正式公文、正式邀请函、报批材料、签证正式材料等政府官方沟通文件，按业务要求使用客户正式名称；脱敏规则主要约束发函前询资源、询接待、询价和第三方协作沟通。

推荐口径：

| 原始客户类型 | 对外表达 |
| --- | --- |
| 具体开发区/园区 | 江苏某国家级开发区、江浙沪某国家级开发区、长三角某国家级开发区 |
| 具体政府部门 | 华东某地方政府部门、江浙沪某政府产业部门、长三角某投促部门 |
| 具体投资局/商务局 | 华东某政府招商机构、江浙沪某投促机构 |
| 具体国企/事业单位 | 华东某国有单位、长三角某事业单位 |

对外版本末尾统一标注：`客户名称已脱敏，正式披露前需由项目负责人确认。`

## 二、核心协作规则

### 2.1 基本原则

> **每个人只在自己的编号文件夹内编辑文件。**
> 不修改他人文件夹内的任何文件。
> 如需讨论或补充，在对方文件夹内新建笔记（标注 `反馈：` 前缀）。

```
项目部/
├── README.md                   ← 本文档（协作总纲）
├── 01-姚玉丽/                   ← 每个人只在自己的文件夹里编辑
│   ├── 姚玉丽工作日志.md
│   └── 2026-05-XX-某项目经验分享.md
├── 02-陈晓蝶/
│   ├── 陈晓蝶工作日志.md
│   └── ...
├── 03-王书洋/
├── 04-塞尔瓦尔/
├── 05-李欣悦/
└── 09-田野/
```

- 编号按入职/加入顺序固定，不随人员变动重排
- 每人仅在自己的文件夹内增删改文件
- **如需给他人的笔记提建议**：在自己的文件夹里新建 `反馈-XX.md`，不要直接修改对方的文件
- **禁止**在 `项目部/` 根目录下散落文件
- 项目级文档一律放 `raw/sources/项目部/` 对应子目录

### 2.2 个人空间：异步沟通 + 经验分享

每个个人文件夹是 **异步沟通空间** 和 **经验分享库**，不是单纯的日志本。

#### ① 工作日志（记录日常）
记录当日工作进展，保持简洁即可：

```markdown
# 工作日志

## [2026-05-09]
- 项目A：完成航班方案提交
- 项目B：对接客户确认行程
- TODO：明日跟进酒店确认函
```

#### ② 经验分享（核心价值）
做完一个项目后，把有价值的认知沉淀下来。这是 **部门共同成长的资产**：

```markdown
# [项目名称] 经验小结

## 场景
简短描述遇到的问题场景。

## 做法
我是怎么处理的。

## 踩坑/教训
哪些地方差点出错，下次怎么避免。

## 可复用的 Tips
下次做类似项目可以直接参考的点。

## 相关链接
- [[wiki/summaries/SOP/XX]]
- [[raw/sources/项目部/历史积累/XX]]
```

经验分享的价值：
- **同一类项目** → 第二个人直接复用第一个人的经验
- **新人上手** → 看大家的分享比看 SOP 更快理解实际执行
- **团队互信** → 看到同事的思考过程，减少信息黑箱

#### ③ 异步沟通（部门内部交流）
把文字交流留在笔记里，减少"口头说完就忘"：

- **有新想法/建议** → 在自己文件夹新建 `想法-XXX.md`
- **想反馈给特定同事** → 在自己文件夹新建 `给XX的建议-XXX.md`
- **需要同步进度** → 在工作日志中标注，@同事来看
- **经验分享后** → 欢迎其他人用 `反馈-XX.md` 补充自己的类似经历

> **异步沟通的好处**：文字留下来就是知识资产，新人能回溯，不需要每次都"口口相传"。每个人都可以按自己的节奏阅读和回复。

## 三、知识库分层（"三明治"架构）

整个 vault 分为三层，理解和遵守这个分层是协作的基础：

| 层级 | 位置 | 谁维护 | 规则 |
|------|------|--------|------|
| **Raw Sources** | `raw/sources/项目部/` | 所有人可写 | 原始文件不可变，LLM只读不修改 |
| **SOP** | `wiki/summaries/SOP/` | 项目部集体维护 | 所有方案必须基于SOP输出 |
| **Knowledge** | `raw/sources/项目部/知识/` | 谁产出谁归档 | 项目完成后整理有价值内容入库 |
| **Historical** | `raw/sources/项目部/历史积累/` | 谁执行谁更新 | 酒店、餐厅、导游报表实时更新 |
| **Wiki** | `wiki/` | LLM维护 | 自动编译的持久知识制品 |
| **项目部** | `项目部/` | 每位成员（仅自己文件夹） | 个人异步沟通 + 经验分享 + 工作日志 |

### 3.1 三个关键动作

#### ① 做方案前 → **先查历史**
- 查 [[项目报表-2026-酒店]] 看同事用过哪些酒店
- 查 [[项目报表-2026-餐厅]] 看哪些餐厅已验证
- 查 [[项目报表-2026-导游]] 看哪些导游合作过
- 三公经费 → 先查 [[wiki/summaries/SOP/三公经费-2017]]，没有再回退到 [[wiki/summaries/SOP/三公经费-2013]]

#### ② 做方案时 → **严格遵循 SOP**
- 航班 → [[01-航班选择]]
- 酒店 → [[02-酒店选择]]
- 报批行程 → [[03-报批行程]]
- 实际行程 → [[04-实际行程]]
- 境外公务 → [[05-境外公务]]
- 餐厅 → [[06-餐厅]]
- 景点 → [[07-景点]]
- 政府公文 → [[09-政府公文]]

#### ③ 项目结束后 → **更新历史数据**
- 把用过的酒店写入 [[项目报表-2026-酒店]]（含评分、价格、联系人）
- 把用过的餐厅写入 [[项目报表-2026-餐厅]]
- 把用过的导游写入 [[项目报表-2026-导游]]
- 把有价值的新认知写入 [[raw/sources/项目部/知识/]]

## 四、项目执行规范

### 4.1 标准执行流程

```
接单 → 查历史 → 出方案 → 内部审核 → 客户确认 → 执行 → 归档
```

每一步的产出物：
1. **航班方案** → 按 SOP 模板 + 携程/航司官网实时查询
2. **酒店方案** → Booking/携程/维途交叉比价，三公经费合规
3. **行程表** → 报批版（用词规范）+ 实际版（精确到分）
4. **出行须知** → 含12项必备内容（见 SOP-04）

### 4.2 核心红线

| 红线 | 说明 |
|------|------|
| 🚫 **不胡编乱造** | 航班价格、酒店评分、餐厅信息等必须来自真实查询 |
| 🚫 **不跳过 SOP** | 每次方案输出严格 SOP 模板，不能省略字段 |
| 🚫 **不对外暴露客户名** | 发给境外/地接/付款方时，脱敏处理客户信息 |
| 🚫 **不把敏感信息给 AI** | 政府客户敏感信息不能使用 AI 工具处理 |
| 🚫 **不改 .obsidian 配置** | 插件和设置由管理员维护，团队成员不修改 |

### 4.3 三公经费合规

- **住宿费**：一人一间，按2017版标准执行，超标200-300元以内可申请说明
- **伙食费 + 公杂费**：按定额标准发放，不需发票
- **国际旅费**：首选国内航司、经济舱、直飞

### 4.4 质量要求

- 所有方案提供 2-3 个备选，标注推荐理由
- 行程必须"全过程模拟"——从客户离开家门到回到家门
- 国际航班至少预留3小时到机场
- 境外大型活动（展会、论坛、节假日）必须在方案中排查说明
- 航班路途安排在周六/周日，确保公务落在工作日上午到下午

### 4.5 知识库内容质量标准

知识库是团队的核心知识资产，入库内容必须通过以下质量审查：

#### ① 真实性——每条信息可追溯
- 所有价格、评分、联系方式、日期等数据必须来自 **真实查询**
- 严禁凭记忆推测、"大概"数据或未经验证的信息
- 信息来源必须在内容中标注（如 `[2026-05-01 查询于携程]`）

#### ② 准确性——宁缺毋滥
- 不确定的信息**必须标注**"待确认"，严禁以确定性语气写入
- 发现错误立即更正，并在修改处标注 `[YYYY-MM-DD] 更新人：XXX`
- 对于可能引起歧义的信息，补充说明上下文

#### ③ 时效性——过时即标注
- 动态数据（价格、评分、政策等）超过30天应重新核实
- 已过时的 SOP 或政策文件必须标注"⚠️ 已过时"并链接到最新版本
- 方案输出时，优先使用最近90天内的历史数据

#### ④ 完整性——无碎片信息
- 每条入库信息应包含足够的上下文：时间、来源、使用场景
- 不收录"半句话"或脱离上下文无法独立理解的信息片段
- 经验分享应包含背景、做法、教训、可复用建议四个要素

#### ⑤ 一致性——消除内部矛盾
- 同一实体（酒店/餐厅/客户）的信息在不同页面间必须一致
- 发现矛盾时，更新方在修改处标注 `[YYYY-MM-DD] 更新人：XXX`
- Wiki 页面矛盾由 LLM 在定期 Lint 中检查并修复

### 4.6 入库红线（否决项）

以下情况的内容**严禁入库**，违反视为质量事故：

| 红线 | 示例 |
|------|------|
| 🚫 **无来源** | 凭记忆写的酒店价格、"好像"的航班信息 |
| 🚫 **未核实** | 从第三方听说但未查证的三公经费标准 |
| 🚫 **过期未标** | 2015年的酒店评分未标注查询时间 |
| 🚫 **猜测推断** | "应该可以住这个酒店"类主观判断 |
| 🚫 **不完整** | 只有酒店名没有地址/电话/价格的残缺条目 |

**审核原则**：不确定 → 不录入。等确认了再补。

## 五、团队协作约定

### 5.1 信息同步

**异步优先，同步为辅**：

| 场景                     | 方式                        | 说明                          |
| ---------------------- | ------------------------- | --------------------------- |
| **日常进展**               | 个人工作日志 + 经验分享             | 写在自己的文件夹，文字即资产              |
| **需要讨论/确认**            | 在 Obsidian 笔记中 @同事 + 飞书提醒 | 先写文字再口头确认，避免说完就忘            |
| **复杂项目**（跨多国、多部门、多人协作） | 建立飞书共享文档同步进度              | 不适合在个人笔记中拉扯的复杂信息            |
| **周会**                 | 参与并按时输出会议纪要               | 纪要归档至 `raw/sources/项目部/知识/` |

### 5.2 客户信息的脱敏规则

| 场景          | 做法                   |
| ----------- | -------------------- |
| 给境外地接发操作单   | 不写客户单位，写"6月法国英国团组8人" |
| 给南京西联/地接要资源 | 不写客户全称，写"江苏某国家级开发区拟开展产业交流" |
| 给供应商/酒店/票代询价 | 不写客户全称，写"华东某政府团组/江浙沪某国家级开发区" |
| 给票代出票       | 脱敏后提供名单              |
| 在 AI 工具中问问题 | 不出现客户单位名称和敏感信息       |
| 内部日志/知识库    | 可记录客户名，但注意区分密级       |

### 5.3 文件命名规范

- 文件名：`[日期]-[项目名称]-[内容类型].md`
- 示例：`2026-06-14-法国英国团组-航班方案.md`
- 禁止：中文空格、特殊符号、过长文件名

### 5.4 Wiki 更新机制

项目结束后，如需将经验沉淀为 Wiki 页面：
- 通知 LLM（Claude）处理，按 `wiki/` 结构自动生成
- 人工审核后确认入库
- 更新 [[wiki/index.md]] 和 [[wiki/log.md]]

## 六、Git 同步规范

### 6.1 自动同步
- 每天早上 8:00（北京时间）自动 commit + push 至 GitHub
- 同步脚本位于 `.claude/sync-to-github.sh`
- 不手动操作 git 命令（除非管理员要求）

### 6.2 注意事项
- `.obsidian/`、`.DS_Store`、`.claudian/sessions/` 已在 `.gitignore` 中排除
- 个人工作日志的更新会随自动同步推送到远端
- 如需紧急同步，联系管理员 Tim

### 6.3 首次加入团队
1. 注册 GitHub 账号
2. 将用户名发给 Tim
3. 安装 Git + Obsidian + Node.js
4. `git clone` 仓库到本地
5. 在 Obsidian 中打开本地文件夹

详细步骤见 [[团队协作环境配置指南]]
