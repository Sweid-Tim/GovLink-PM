# GovLink 知识库维基 (Knowledge Wiki)

> 基于 Karpathy 的 LLM Wiki 方法论构建的个人知识库。
> 核心理念：**LLM 负责维护，你负责思考**。

---

## 核心理念

传统 RAG 的问题：每次提问，LLM 从零发现知识，没有积累。

我们的方案：让 LLM **增量构建并维护一个持久的 wiki**。新资料进来时，LLM 读取、提取、整合——更新实体页、修订摘要、标记矛盾、强化联系。知识编译一次，**持续保持最新**，不再每次重复推导。

关键区别：**wiki 是一个持久、复利的知识制品**。交叉引用已就位，矛盾已标记，综合已反映所有读过的东西。

---

## 三层架构

### 1. Raw Sources（原始资料层）
- **位置**: `raw/sources/`
- **作用**: 存放不可变的源文档。LLM 只读不写，这是你的真理来源。
- **内容类型**: 文章、PDF、网页剪藏、微信文章、图片、数据文件等
- **规则**: 原始资料永远是原始的，LLM 不修改它们

### 2. Wiki（维基层）
- **位置**: `wiki/` 及其子文件夹
- **作用**: LLM 生成和维护的结构化笔记。LLM 全权拥有这一层。
- **子文件夹**:
  - `entities/` — 实体页（人物、公司、产品、项目）
  - `concepts/` — 概念页（方法论、术语、原理）
  - `summaries/` — 摘要页（书籍摘要、文章摘要、会议纪要）


### 3. Schema（本文件）
- **位置**: `AGENTS.md`
- **作用**: 告诉 LLM wiki 的结构、约定、以及工作流程
- **维护**: 你和 LLM 共同演进，随着领域理解加深不断完善

## 文件约定

### Wiki 页面命名
- 使用简洁的中文标题，如 `大美政通-组织管理.md`
- 实体页以实体名称开头，如 `大美政通.md`
- 概念页用简洁术语，如 `OKR管理.md`

### Wiki 页面结构
```markdown
---
title: 页面标题
tags: [entity, company]
created: 2026-04-29
sources: 1
---

# 页面标题

## 概述
简短介绍。

## 关键信息
整理的关键点。

## 相关资料
- [[wiki/summaries/某书摘要]]
- [[raw/sources/原始文章]]

## 最近更新
- [2026-04-29] 创建页面
```

## 工作流程

### Ingest（摄入新资料）

当你有新资料要处理时：

1. 将资料放入 `raw/sources/` 或其子分类
2. 告诉 LLM：`请处理这个资料，更新维基`
3. LLM 会：
   - 读取源资料
   - 与你讨论关键要点
   - 在 wiki 中写摘要页
   - 更新 index.md
   - 更新相关实体和概念页
   - 在 log.md 中记录

**单次摄入可能涉及 10-15 个 wiki 页面**。

### Query（查询）

当你想查询知识时：

1. 直接提问，如：`关于公司组织管理有什么沉淀？`
2. LLM 会：
   - 搜索 wiki 中的相关页面
   - 阅读并综合答案
   - 给出带引用的回复
3. **重要**：如果回复有价值，让 LLM 将其保存为 wiki 页面

### Lint（健康检查）

定期（每周或每月）让 LLM 检查 wiki 健康度：

- 页面之间的矛盾
- 被新资料超越的旧观点
- 没有入口链接的孤立页面
- 有提及但没有自己页面的重要概念
- 缺失的交叉引用
- 可以用网络搜索填补的数据空白

## 客户信息脱敏与外发口径（全局红线）

> 凡是生成可能对外发送、转发、复制给第三方的前期沟通内容，必须默认对客户身份做脱敏处理。此规则适用于所有 SOP、会议纪要、邮件草稿、微信/飞书消息、询价需求、地接需求、供应商需求、境外机构联系、发函前沟通和资源开发。

### 必须脱敏的对象

- 政府单位、开发区、园区、投促局、外办、商务局等政府或准政府客户。
- 事业单位、国企、学校、医院等敏感客户。
- 尚未公开的项目名称、团组名称、领导姓名、团员名单、预算、具体出访安排和客户内部偏好。

### 外发口径写法

| 客户原始类型 | 对外脱敏说法 |
|:---|:---|
| 具体开发区/园区 | 江苏某国家级开发区、江浙沪某国家级开发区、长三角某国家级开发区 |
| 具体政府部门 | 华东某地方政府部门、江浙沪某政府产业部门、长三角某投促部门 |
| 具体投资局/商务局 | 华东某政府招商机构、江浙沪某投促机构 |
| 具体国企/事业单位 | 华东某国有单位、长三角某事业单位 |
| 具体协会/园区客户 | 国内某产业园区/行业机构 |

### 禁止事项

- 不得在给南京西联、地接社、供应商、酒店、航司、票代、境外机构、商协会等第三方的口径中直接写客户全称。
- 不得写具体领导姓名、团员名单、内部决策过程、预算细节和客户偏好原话。
- 内部版可以保留客户真实名称用于追踪；对外版必须另行生成脱敏版本。
- 正式公文、正式邀请函、报批材料、签证正式材料等政府官方沟通文件，按业务要求使用客户正式名称；脱敏规则主要约束发函前询资源、询接待、询价和第三方协作沟通。
- 对外版本末尾必须标注：`客户名称已脱敏，正式披露前需由项目负责人确认。`

示例：把“昆山开发区投资局拟于6月下旬赴土耳其考察”改为“江苏某国家级开发区拟于6月下旬赴土耳其方向开展产业交流”。

## 输出规范（Output Standards）

> 当用户要求输出 **航班方案、酒店方案、行程方案** 等出团相关内容时，必须严格遵循以下规范。

### 前置流程（每次执行前）

1. **先读 SOP** — 根据需求类型读取对应的 SOP：
   - 航班 → [[wiki/summaries/SOP/01-航班选择]]
   - 酒店 → [[wiki/summaries/SOP/02-酒店选择]]
   - 报批行程 → [[wiki/summaries/SOP/03-报批行程]]
   - 实际行程 → [[wiki/summaries/SOP/04-实际行程]]
   - 境外公务 → [[wiki/summaries/SOP/05-境外公务]]
   - 三公经费 → [[wiki/summaries/SOP/08-三公经费]] + [[wiki/summaries/SOP/三公经费-2013]] + [[wiki/summaries/SOP/三公经费-2017]]
2. **再查历史数据**：
   - 酒店 → [[raw/sources/项目部/历史积累/酒店-20260507]]
   - 餐厅 → [[raw/sources/项目部/历史积累/餐厅-20260507]]
   - 导游 → [[raw/sources/项目部/历史积累/导游-20260507]]
   - 公务资源 → [[raw/sources/项目部/历史积累/公务资源-20260511]]（779条，按国家+分类双索引）
3. **核查三公经费**：先查2017版标准，没有再回退到2013版
4. **检查大型活动**：确认行程期间境外是否有国际展会、论坛、会议或当地大型节假日

### 航班方案输出格式

**去程/回程/城际段各一张独立表格**，每张表格包含：

| 方案选择 | 航班名称 | 出发城市 | 出发机场/航站楼 | 出发日期/时间 | 抵达城市 | 抵达机场/航站楼 | 抵达日期/时间 | 全程用时 | 行李 | 中转时长 | 参考价格 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|

- 提供 2-3 个方案，第3个标注为"政采航班"
- 首选国内航司、经济舱、直飞
- 默认7天，凌晨出发、第7天24:00前抵达国内
- 在航班号旁标注 🏆 表示国内航司优先
- 中转航班必须满足：免过境签、行李直挂、中转≥1.5h

## 特殊文件

### index.md（维基目录）
- **位置**: `wiki/index.md`
- **作用**: 所有 wiki 页面的目录，每个页面有一行摘要
- **格式**: 按类别组织（实体、概念、摘要等）
- **维护**: 每次 ingest 后更新

### log.md（活动日志）
- **位置**: `维基/log.md`
- **作用**: 按时间顺序记录所有活动（摄入、查询、lint）
- **格式**: 每条记录以 `## [日期] 类型 | 标题` 开头
- **用途**: 追踪 wiki 演化历史，帮助 LLM 理解最近做了什么

**快速查看最近活动**:
```bash
grep "^## \[" wiki/log.md | tail -5
```

## GitHub 自动同步

每天自动将整个 vault 同步至 [GovLink-PM](https://github.com/Sweid-Tim/GovLink-PM) 仓库。

### 同步方式
- **本地定时任务**: macOS launchd 每天北京时间 8:00 自动 commit + push
- **同步范围**: 整个 vault（含 raw/、wiki/、笔记文件夹等）
- **认证方式**: SSH Key

### 本地脚本
同步脚本位于 `.Codex/sync-to-github.sh`，内容包括：
1. `git add -A` 暂存所有变更
2. `git commit`（若无变更则跳过）
3. `git push` 推送到 GitHub

### Git 配置
- `.gitignore` 排除 `.obsidian/`、`.DS_Store`、`.claudian/sessions/` 等本地配置

## 工具推荐

- **Obsidian Web Clipper**: 浏览器插件，将网页转换为 markdown
- **图片本地化**: 设置 → 文件和链接 → 附件文件夹路径设为 `raw/assets/`，然后用热键 `Ctrl+Shift+D` 下载图片
- **Graph View**: 查看 wiki 的结构——哪些页面互联、哪些是孤立页面
- **Dataview**: 用 frontmatter 生成动态表格和列表
- **Marp**: 从 wiki 内容生成演示文稿
- **Claudian**: Obsidian 的 AI 协作者插件，直接在 vault 内运行 Codex

## 为什么这有效

维护知识库最繁琐的不是阅读或思考，而是 ** bookkeeping（记账式工作）**：
- 更新交叉引用
- 保持摘要最新
- 标记矛盾
- 维护一致性

人类放弃 wiki 是因为维护负担增长快于价值。LLM 不会厌倦、不会忘记更新交叉引用、一次可以修改 15 个文件。

**人的工作是**：策划来源、引导分析、问好问题、思考意义。

**LLM 的工作是**：其他一切。
